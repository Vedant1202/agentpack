    Spec: Parsing & Retrieval Overhaul

    Status: SPEC APPROVED (Phase 1 ✓) — Implementation Plan below awaiting review (Phase 2 gate) Owner: Vedant Created: 2026-05-29 Supersedes parts of: docs/architecture.md

    Objective

    AgentPack compiles documents into an agent-friendly, citation-first context pack and serves retrieval over it. Two reviews surfaced a set of correctness, quality, and performance gaps across the parsing pipeline (especially the Docling PDF path) and the retrieval layer.

    This spec defines the work to make AgentPack:

    1. Trustworthy — retrieval never silently serves stale or wrong-citation results.
    2. High-fidelity — semantic parsing preserves page numbers and tables, not just plain text.
    3. Fast at scale — packing and querying stay usable on corpora of thousands of files/chunks.

    Who is the user

    Engineers and AI builders who pack a corpus of PDFs/docs and let an agent retrieve cited evidence from it. Correct citations (source + page + section) are the core value proposition.

    What success looks like (top-level, testable)

    • Re-packing a corpus into an existing output dir never serves results from the previous corpus.
    • A PDF parsed in semantic (Docling) mode yields chunks with correct page citations and type="table" blocks for tabular content — parity with or better than --fast-pdf.
    • No chunk exceeds the configured max_tokens.
    • Re-packing a corpus where only 1 of N files changed reprocesses ~1 file, not N.
    • The embedding model and Docling converter are each loaded at most once per process.
    • The full parser + retrieval test suite passes with Docling installed, in CI.

    Tech Stack

    • Python >=3.9, pydantic>=2, typer CLI
    • Parsing: docling>=2 (semantic), pymupdf (fast/spatial fallback), pandas/tabulate (CSV)
    • Retrieval: sqlite3 FTS5 (lexical), fastembed (ONNX/CPU embeddings), numpy<2 (vectors)
    • Vector backend: hnswlib (HNSW, default — core dependency), sqlite-vec (exact, used in --fast mode)
    • Tokenization: tiktoken (cl100k_base)
    • Optional UI: fastapi + uvicorn
    • Tests: pytest, pytest-mock

    Commands

    Install (with semantic + test extras): pip install -e ".[test]"   # docling is a core dep
Build pack (full/semantic):            agentpack pack <input_dir> --out <out_dir>
Build pack (fast/lightweight):         agentpack pack <input_dir> --out <out_dir> --fast
                                       # --fast = PyMuPDF parse + sqlite-vec; default = Docling + HNSW
                                       # --fast-pdf kept as a deprecated alias for --fast
Pre-build indexes:                     agentpack index <pack_dir>        # NEW command
Retrieve:                              agentpack retrieve <pack_dir> "<query>" --mode hybrid
Validate:                              agentpack validate <pack_dir>
Audit:                                 agentpack audit <pack_dir>
Test (all):                            python -m pytest -q
Test (parsers only):                   python -m pytest tests/test_parsers.py -q

    Project Structure

    src/agentpack/
  parsers/        → Parser per file type (text, markdown, csv, pdf). New: docx/pptx/xlsx/html.
  pack.py         → Orchestration: scan → parse → chunk → write. Add incremental cache + parallelism.
  chunker.py      → Block→chunk tokenization. Add intra-block splitting.
  retrieve.py     → FTS / vector / hybrid search + index build. Add invalidation, RRF, singletons.
  models.py       → Canonical Document Model (SourceDocument / DocumentBlock / ExtractionWarning).
  cli.py          → Typer commands. Add `index`; wire `--config`.
  ui/server.py    → FastAPI corpus explorer.
docs/specs/       → This spec and future specs (committed alongside code).
tests/            → pytest suite. Tests live here, mirroring module names (test_parsers.py, ...).

    Code Style

    Match existing style: explicit pydantic models, small functions, typed signatures, warnings collected as data (not raised) during parsing. Example of the target shape for the Docling path consuming the structured tree (not a markdown round-trip):

    def _parse_semantic(self, file_path: Path, source_id: str) -> SourceDocument:
    result = self._converter().convert(file_path)          # cached converter
    blocks: list[DocumentBlock] = []
    for item, _level in result.document.iterate_items():
        page = _page_of(item)                              # preserve provenance
        if item.label == "table":
            blocks.append(DocumentBlock(
                block_id=f"{source_id}_b{len(blocks)}", source_id=source_id,
                type="table", text=item.export_to_markdown(), page=page,
            ))
        elif item.label in ("section_header", "title"):
            blocks.append(DocumentBlock(..., type="heading", text=item.text, page=page))
        elif item.text:
            blocks.append(DocumentBlock(..., type="paragraph", text=item.text, page=page))
    return SourceDocument(source_id=source_id, path=file_path.name, type="pdf", ...)

    Conventions: block_id = f"{source_id}_b{n}"; warnings use ExtractionWarning(type=..., message=...); never print from library code paths except behind verbose and not quiet.

    Testing Strategy

    • Framework: pytest, fixtures in tests/conftest.py, mocking via pytest-mock / unittest.mock.
    • Location: tests/, one module per source module.
    • Levels:
    • Unit — parsers (per type), chunker splitting, RRF fusion math, index-hash invalidation logic.
    • Integration — full pack → index → retrieve round-trip on a small fixture corpus (1 PDF with a table + 1 md + 1 csv), asserting page/section/table citations survive end-to-end.
    • Docling in CI: semantic-path tests must run with Docling actually installed (no skip). Use a tiny committed fixture PDF, not a mocked converter, for at least one end-to-end test.
    • Regression guards: a test that re-packs into an existing dir and asserts the index is rebuilt (no stale results); a test asserting no chunk exceeds max_tokens.
    • Coverage expectation: every fixed gap below ships with at least one failing-then-passing test.

    Boundaries

    • Always: run python -m pytest -q before committing; keep --fast-pdf working as a fallback; collect parse failures as ExtractionWarnings rather than crashing the whole pack; keep CLI flags backward compatible.
    • Ask first: adding heavyweight deps (faiss/torch/rerank models); changing the on-disk pack format or manifest schema; changing default max_tokens/embedding model; making Docling a hard failure when missing.
    • Never: commit secrets or large binary corpora; serve a retrieval result without re-validating the index against the current pack; remove or xfail a failing test to make CI green without approval; delete the demo_corpus/, src/test_output_* scratch dirs without confirming.

    Scope: Gaps & Success Criteria

    Each item is independently testable. IDs are referenced by the Plan/Tasks phases.

    A. Docling / parsing fidelity

    • G-A1 Docling installed & exercised in CI. Semantic test runs against a real (tiny) PDF; pip install -e ".[test]" yields an importable docling. Success: test_pdf_parser_semantic passes without mocking the converter.
    • G-A2 Preserve page numbers in semantic mode. Consume Docling's structured doc; every block carries its source page. Success: semantic-mode chunks expose citation.page, matching the page a string appears on in the fixture PDF.
    • G-A3 Preserve tables as type="table". Tables become table blocks (written to tables/ and recorded in the manifest), not paragraphs. Success: fixture PDF's table appears in tables/*.csv|md and in manifest.tables, and is excluded from overlap duplication.
    • G-A4 Reuse one converter. DocumentConverter built once per process, not per file. Success: a multi-PDF pack constructs the converter exactly once (assert via spy/log).
    • G-A5 Robust semantic test. Replace the brittle module-patch test with a fixture-based one.

    B. Retrieval correctness

    • G-B1 Index invalidation. FTS + vector indexes store the manifest content hash; a changed pack triggers rebuild. Success: re-pack with new content → next retrieve reflects new content.
    • G-B2 Explicit agentpack index command. Builds both indexes; pack optionally invokes it. Success: index exists, is idempotent, and first post-index query pays no build cost.

    C. Chunking

    • G-C1 Split oversized blocks. Any block > max_tokens is split (with overlap) so no chunk exceeds the limit. Success: a synthetic 5k-token block yields multiple ≤max_tokens chunks.

    D. Retrieval quality

    • G-D1 RRF fusion replaces min-max hybrid scoring. Success: fusion is rank-based; unit test on known rankings matches expected RRF order.
    • G-D2 FTS precision. Move off pure OR-of-terms (phrase/AND/BM25 weighting). Success: a multi-term query ranks chunks containing all terms above chunks containing one.
    • G-D3 Metadata filtering. Retrieve can filter by source_id / page / section. Success: --source/--section flags constrain results.
    • G-D4 Reranker hook — optional cross-encoder rerank pass behind a flag/extra. DEFERRED to v2 spec.

    E. Performance

    • G-E1 Embedding model singleton (build + query share one instance). Success: model loaded once.
    • G-E2 Pre-normalized vectors at index build; query cosine = dot product. Success: no per-query linalg.norm over the matrix; results unchanged within tolerance. Remains the fallback path when no vector extension is available / for tiny corpora.
    • G-E3 Incremental packing keyed by source checksum; unchanged files reuse cached blocks/chunks. Success: re-pack with 1/N changed reprocesses ~1 file. (Realized via the L1 parse cache, G-H1.)
    • G-E4 Parallel parsing across files (process pool), order-stable output. Success: wall-clock speedup on multi-file corpus; identical manifest vs sequential.
    • G-E5 ANN vector index (default). Replace brute-force np.dot with an HNSW index (hnswlib) as the default vector backend — true approximate NN, fast queries at scale. The pre-normalized brute-force path (G-E2) stays as the no-extension fallback. Success: vector search on a ~10k-chunk corpus returns top-k via HNSW with >0.95 recall@10 vs exact.
    • G-E6 One lightweight switch, no backend flag. Default (full/semantic) mode = Docling + HNSW. A single --fast mode = PyMuPDF parse + sqlite-vec (exact, in-SQLite, ~instant index build, minimal deps) — ideal for small corpora / quick iteration. Generalize the existing --fast-pdf into --fast (keep --fast-pdf as a deprecated alias) so users face one mental switch, not a separate vector-backend option. Success: --fast selects PyMuPDF + sqlite-vec; default selects Docling + HNSW; no standalone vector-backend flag exists.
    • G-E7 Baseline benchmark harness. A small, repeatable script measuring cold-pack time, re-pack time, and query p50/recall across a few corpus sizes. Establishes the baseline that perf work is judged "measurably faster" against, and the evidence for any future numeric gates. Success: benchmarks/ harness runs and prints a before/after comparison table.

    F. System design / data model

    • G-F1 Manifest version correctness — single source of version (read from package), not 0.1.0.
    • G-F2 SQLite as chunk-metadata source of truth; YAML manifest becomes a lean summary. (May be staged; requires format-change approval.)
    • G-F3 Chunk storage consolidation — evaluate JSONL/SQLite over thousands of tiny .md files.

    G. Product surface

    • G-G1 Broader formats via Docling: docx, pptx, xlsx, html (+ images/OCR toggle). Incremental, one parser registration each.
    • G-G2 Character-offset provenance back into source (enables highlight-in-original). DEFERRED to v2 spec.
    • G-G3 Secret/PII redaction — confirm/wire detect-secrets into the scan/parse path; redact or flag before packing. Success: a planted secret in a fixture is flagged/redacted in output.
    • G-G4 Config file (agentpack.toml: chunk size, embedding model, included types) for reproducible packs.

    H. Caching (content-addressed, layered)

    Re-runs should pay only for what changed. Adopt the LlamaIndex IngestionPipeline / LangChain CacheBackedEmbeddings pattern: hash each transformation's input and key its output by that hash. AgentPack already computes sha256(content) per source (pack.py:111) but discards it — that is the cache key. Store layers in a single cache.db (SQLite) alongside indexes/. Keys embed a version component so changing logic auto-invalidates only the affected layer.

    • G-H1 Parse cache (L1). Key sha256(file_bytes) + parser_version + parse_opts → serialized SourceDocument. Skips Docling entirely on unchanged files. (This is how G-E3 is realized.) Success: re-pack of an unchanged corpus performs zero Docling conversions.
    • G-H2 Embedding cache (L3). Key sha256(chunk_text) + model_id → vector BLOB. Unchanged chunks are never re-embedded, even when their file changed elsewhere. Success: editing one paragraph in a large file re-embeds only the affected chunk(s).
    • G-H3 Index invalidation (L4). Same mechanism as G-B1 — pack content hash stamped in each index; mismatch triggers rebuild. Listed here to anchor it in the cache-coherence model.
    • G-H4 (Optional) Query cache (L5). Key sha256(query + mode + top_k + pack_hash) → result set. Memoizes repeated/UI queries. Success: an identical repeat query is served without re-search.
    • G-H5 Versioned keys. parser_version / chunker_version / model_id participate in keys so a logic change self-invalidates without a manual cache clear. Success: bumping chunker_version forces re-chunk but reuses the L1 parse cache (no re-parse).

    Success Criteria (rollup — definition of done)

    • [ ] Semantic PDF parse preserves page + tables (G-A2, G-A3); CI runs it for real (G-A1).
    • [ ] No stale-index results possible (G-B1); agentpack index exists (G-B2).
    • [ ] No chunk exceeds max_tokens (G-C1).
    • [ ] Hybrid uses RRF (G-D1); heavy resources loaded once (G-A4, G-E1); vectors pre-normalized (G-E2).
    • [ ] Incremental + parallel packing measurably faster on a multi-file corpus (G-E3, G-E4).
    • [ ] HNSW is the default vector backend; --fast selects PyMuPDF + sqlite-vec (G-E5, G-E6).
    • [ ] Content-addressed cache: unchanged files skip Docling; unchanged chunks skip embedding (G-H1, G-H2).
    • [ ] Manifest version is correct and tables are recorded (G-F1, G-A3).
    • [ ] Full test suite green in CI with Docling installed.

    3. Redaction (G-G3): is detect-secrets meant to redact content or only flag it in warnings?
    4. New formats (G-G1): in-scope for this spec, or split into a follow-up spec once the core correctness/perf work lands?

    Phasing (ordered by dependency, not importance)

    Phase 0 — Foundations & safety net (do first; unblocks everything; mostly sequential) - Pin/verify Docling installed; convert the brittle semantic test to a real-fixture test (G-A1, G-A5). - Add the integration fixture corpus (1 PDF w/ table + 1 md + 1 csv) and a pack→index→retrieve round-trip test that currently fails on citations — our regression net. - Build the baseline benchmark harness (G-E7) and record current numbers. - Checkpoint: CI runs Docling for real; baseline table committed.

    Phase 1 — Correctness (highest severity; can parallelize the three tracks) - Track 1.1 Docling fidelity: consume structured doc tree → preserve page, emit type="table" blocks; reuse one converter (G-A2, G-A3, G-A4). Manifest records tables; version fix (G-A3, G-F1). - Track 1.2 Index invalidation: stamp pack content hash in FTS + vector indexes; rebuild on mismatch (G-B1). Add agentpack index command (G-B2). - Track 1.3 Chunking: split oversized blocks with overlap (G-C1). - Checkpoint: the Phase-0 round-trip test now passes (pages + tables cited end-to-end); no chunk exceeds max_tokens; re-pack never serves stale results.

    Phase 2 — Caching & build performance (depends on Phase 1 stable models) - L1 parse cache → realizes incremental packing (G-H1, G-E3). Versioned keys (G-H5). - L3 embedding cache (G-H2). Single cache.db. - Parallel parsing across files (G-E4). - Checkpoint: re-pack of unchanged corpus does zero Docling conversions; benchmark shows the win.

    Phase 3 — Retrieval quality & query performance (parallelizable) - Embedding-model singleton + pre-normalized vectors (G-E1, G-E2). - HNSW default backend via hnswlib; brute-force fallback retained (G-E5). - --fast mode = PyMuPDF + sqlite-vec; deprecate --fast-pdf alias (G-E6). - RRF fusion replaces min-max (G-D1); FTS precision AND/BM25 — pending Q3 decision (G-D2); metadata filtering by source/page/section (G-D3). Optional query cache (G-H4). - Checkpoint: recall@10 >0.95 vs exact on the ~10k fixture; hybrid ordering verified by unit test.

    Phase 4 — Product surface (lowest coupling; do last / can defer per-item) - Config file agentpack.toml (G-G4). - Secret/PII handling wired into scan (G-G3). - New formats docx/pptx/xlsx/html via Docling (G-G1). - SQLite-as-source-of-truth + chunk-storage consolidation (G-F2, G-F3) — pending Q1/Q2 format decisions; may split to its own spec.

    Risks & mitigations

    • Docling structured-tree API mismatch (page extraction varies by version) → pin a Docling version; cover with the real-fixture test before refactoring.
    • HNSW non-determinism (ANN recall varies) → keep brute-force as ground truth; assert recall threshold, not exact equality.
    • Cache key bugs = stale output (the exact class of bug we're fixing) → versioned keys + a cache-invalidation test are themselves acceptance criteria.
    • --fast rename breaks existing scripts → keep --fast-pdf as a working deprecated alias.
    • Format-change scope creep (Phase 4) → gated behind Q1/Q2; can fork to a v2 spec.

    Parallelism summary

    • Sequential gates: Phase 0 → Phase 1 → Phase 2. (Each builds on the prior's stable models.)
    • Parallel within Phase 1 (3 tracks) and Phase 3.
    • Phase 4 items are mutually independent and individually deferrable.

    Still needs a decision before Phase 3

    • Q2 (FTS ranking, G-D2): change default ranking to AND/BM25 (reorders results) or keep OR + only add RRF on top? Everything else in Phase 3 is unaffected.
    • Q1 (format change, G-F2/F3): in this spec or a follow-up? Only gates Phase 4. ```
