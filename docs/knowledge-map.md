# Hierarchical Knowledge Map (`map.yml`)

Alongside `manifest.yml`, every pack emits a **`map.yml`** — a compact, hierarchical map of *what information lives where* in the corpus. It is built for **agentic / RAG-style retrieval**: an agent reads the map to reason about where to look, then pulls only the chunks it needs.

Where `manifest.yml` is a flat registry of chunks and citations, `map.yml` is a navigable tree:

```text
corpus → document → section (recursive) → chunk_ids
```

It is generated by default during `agentpack pack` (disable with `--no-map`).

## Why a map?

The flat manifest tells an agent a chunk's *address* (`page 11`), but not its *aboutness* or the document's *structure*. To answer "where does this corpus discuss restructuring liabilities?" an agent would otherwise have to read chunks. The map encodes the section hierarchy AgentPack already parses (headings, page ranges) plus deterministic descriptors, so an agent can drill **top-down** — corpus → document → section → chunk — without scanning the whole pack.

## Structure

```yaml
map_version: 1
pack: { name, generated_at, manifest: manifest.yml }
corpus:
  summary: "..."                      # extractive (TextRank) overview
  stats: { documents, sections, tables, chunks }
documents:
- source_id: src_000                  # → manifest.sources[].id
  path: report.pdf
  title: "Annual Report"
  status: success                     # "failed" if the source did not parse
  pages: [1, 188]
  summary: "..."                      # extractive document summary
  stats: { sections, tables, chunks }
  sections:                           # recursive TOC tree
  - node_id: src_000_s07
    title: "Consolidated Balance Sheets"
    pages: [96, 101]
    has_tables: true
    keyphrases: [accounts receivable, inventory, current liabilities]
    gist: "Current assets incl. net trade receivables; current liabilities."
    chunk_ids: [src_000_chunk_241, src_000_chunk_242]
    nodes: []                         # child sections (same shape, recursive)
```

| Tier | Fields |
|---|---|
| **corpus** | `summary` (extractive), `stats` |
| **document** | `title`, `status`, `pages`, `summary`, `stats`, `sections` |
| **section** (recursive) | `node_id`, `title`, `pages`, `has_tables`, `keyphrases`, `gist`, `chunk_ids`, `nodes` |

- **Structure** (`node_id`, `title`, `pages`, `has_tables`, `chunk_ids`, `nodes`) is reconstructed deterministically from the parsed `DocumentBlock.section_path` + page numbers — so sections whose prose merged into a neighbouring chunk are still represented.
- **`chunk_ids`** at every node point back to `manifest.chunks`. A document's chunks attach to the section matching their `section_path`; chunks with no section land under a synthetic `__root__` node.
- **`status: failed`** flags sources that did not parse / produced no chunks — they are kept (with empty sections) so the agent sees the full corpus.
- **`node_id`** is a stable, deterministic ordinal path (e.g. `src_000_s07-02`).

## Descriptors are deterministic and offline

The `keyphrases`, `gist`, and `summary` fields are produced with **no LLM and no network**:

- **`keyphrases`** — [YAKE](https://github.com/INESCTEC/yake): unsupervised, language-independent, statistical keyphrase extraction (no training, dictionaries, or corpora).
- **`gist` / `summary`** — extractive **TextRank** (PageRank over sentence overlap, via `networkx`): a verbatim, most-central sentence.

Both are **domain-agnostic** (no finance/legal/etc. assumptions) and **deterministic** — the same corpus always yields the same map. The topic signal lives only at the **section** tier (`keyphrases`); synthesized document/corpus topic *lists* are intentionally omitted, because rolling many sections into one list biases toward whatever repeats (e.g. page headers). Documents and the corpus carry only an extractive `summary`.

## The map never affects retrieval

`map.yml` is a **navigation layer, not an index**. Its descriptors are never written into the FTS or vector index (those are built purely from chunk *content*), and the retrieval engine never reads the map. Enabling, disabling, or regenerating the map therefore cannot change what `agentpack retrieve` returns — it only changes what an agent sees when deciding *where* to look.

## CLI

```bash
agentpack pack ./docs --out ./out               # map.yml built by default
agentpack pack ./docs --out ./out --no-map      # skip the map
agentpack map ./out                             # (re)build map.yml for an existing pack
```

`agentpack map` rebuilds the map from an existing pack's manifest alone (no re-parse). It is a lighter-weight reconstruction — it cannot recover `has_tables`, sections with no chunks, or descriptors; run `agentpack pack` for the full-fidelity map.

`agentpack validate` also checks `map.yml` when present (every `chunk_ids` / `source_id` resolves against the manifest); an absent map is not an error.
